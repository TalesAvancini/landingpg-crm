  # Plan: API Notion CRM Integration

## 📌 Overview
This plan outlines the design and implementation of the Serverless API and Notion CRM integration for the feature `api_notion_crm`. The system qualifies and centralizes leads by ingesting webhook payloads from a Typebot conversational form and inserting them directly into a structured Notion database. Notion serves as the exclusive commercial database and Single Source of Truth (SSOT).

The main components include:
1. **Notion Database Configuration (`scripts/setup-crm.js`)**: A CLI script to automatically set up the Notion database under a designated parent page.
2. **Serverless Webhook Intake (`api/webhook-intake.js`)**: An endpoint to receive lead data, format phone numbers into clickable WhatsApp links, perform retry logic with exponential backoff, and insert leads into the database.
3. **Authentication & Validation**: Bearer token authentication for webhook security and input validation.

---

## 🏗️ Project Type
`BACKEND` / `WEB` (Vercel Serverless Functions and Node.js backend integration scripts)

---

## 🎯 Success Criteria
* **Database Setup**: Running `node scripts/setup-crm.js` successfully creates a Notion Database inside the parent page (`NOTION_PAGE_ID`) with the correct properties: Name, Company, WhatsApp (URL format), Email, Budget (Select), Status (Select), and returns the created Database ID.
* **Webhook Security**: The webhook rejects requests that lack a valid Bearer token in the `Authorization` header matching `TYPEBOT_WEBHOOK_SECRET` (returns `401 Unauthorized`).
* **Validation**: Webhook validates that payloads contain `name`, `company`, `whatsapp`, `email`, and `budget` in the correct formats, returning `400 Bad Request` if validation fails.
* **WhatsApp Formatting**: Phone numbers are processed to extract digits only and formatted into a clickable WhatsApp link (`https://wa.me/{number}`) before insertion.
* **Resilience**: The webhook performs up to 3 retries in-memory with exponential backoff on Notion API failures. If all fail, it returns a `500 Internal Server Error` or `503 Service Unavailable`.
* **Latency**: The ingest process completes under 3 seconds under normal network conditions.
* **Architecture Integrity**: No SQL or NoSQL database dependencies are introduced. Notion remains the exclusive database (SSOT).

### Task 3: Webhook Ingestion Endpoint
* **Agent**: `@backend-engineer`
* **Skill**: `tlc-spec-driven`, `coding-guidelines`
* **INPUT**: `.env` (containing `NOTION_TOKEN`, `NOTION_DATABASE_ID`, and `TYPEBOT_WEBHOOK_SECRET`)
* **OUTPUT**: `api/webhook-intake.js`
* **Description**:
  1. Create a Vercel-compatible serverless Node.js handler.
  2. Implement validation for the `Authorization` header: must start with `Bearer ` and match `TYPEBOT_WEBHOOK_SECRET`. Return `401 Unauthorized` if invalid.
  3. Validate the payload body to ensure that `name`, `company`, `whatsapp`, `email`, and `budget` are present and properly typed. Return `400 Bad Request` on failure.
  4. Implement phone number cleaning: extract digits only from `whatsapp`, check for minimum/maximum length, and format it as `https://wa.me/{digits}`.
  5. Implement database insertion logic using `@notionhq/client` to insert a page in `NOTION_DATABASE_ID` with status set to `Triagem (Typebot)`.
  6. Add resilience: Wrap the database call in a retry loop (3 retries max) with exponential backoff (e.g., initial delay of 500ms, backing off to 1000ms, then 2000ms).
  7. Return `201 Created` or `200 OK` on success, or `500` / `503` if all database retries fail.
* **VERIFY**:
  - Code review validates that the Bearer token matches `TYPEBOT_WEBHOOK_SECRET`.
  - Code review confirms that no PostgreSQL, MongoDB, or third-party workflow tool (Make/Zapier) dependencies are introduced.
  - In-memory backoff function exists and correctly increments delay.
